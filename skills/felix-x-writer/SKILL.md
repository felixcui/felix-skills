# 文章转 X (Twitter) 动态

把一篇本地 Markdown 文章，浓缩成 1–3 条真诚、克制、有观点的中文 X 动态。**默认优先用单条**——尽量把想说的核心内容都压进一条、把 280 额度用足；只有单条实在装不下时才拆成 thread（多条连发，最多 3 条）。每条都用脚本校验，确保压在 X 的 280 字符限制之内。

风格定位是**真诚说理型**——不是标题党，不是钩子套路，而是把文章里真正有价值的观点，用人话、有判断地讲出来，接近公众号那种"简明、有深度、不浮夸"的调性，但比公众号更短更冲。

## 参数

- `$ARGUMENTS`：本地 Markdown 文件路径（必填），以及可选的输出目录路径
  - 例：`felix-x-writer ~/Desktop/my-post.md`
  - 例：`felix-x-writer ~/Desktop/my-post.md --output-dir ~/Desktop/x`

## 字数规则（这是本 skill 的技术核心，务必吃透）

X 的字符限制是 **280 权重**，但不是"280 个字"——它按字符类型加权，中文比英文"贵"：

| 字符类型 | 权重 | 说明 |
|---------|------|------|
| ASCII（英文字母、数字、英文标点、空格） | 1 | `a-z` `0-9` `,.` `@` `#` 等都算 1 |
| 中文 / 日文 / 韩文 | 2 | 每个中文字算 2 权重 |
| emoji、全角标点、箭头等符号 | 2 | ✅🔥→ 等都算 2 |
| URL（`http://` 或 `https://` 开头） | 23 | 无论实际多长，X 会用 t.co 包装，固定算 23 |

**中文单条容量换算**：
- 纯中文无链接：约 **140 个中文字**（280 ÷ 2）
- 带 1 个链接：约 **128 个中文字**（先扣 23 链接 + 1 空格）
- thread 里**每一条都独立按 280 计算**，不是总和

为什么这条这么重要：很多人凭"感觉"写完 160 个中文字，以为没超，实际权重已经 320 了，X 上根本发不出去。所以本 skill **强制用脚本校验**，不靠模型目测。

## 输出目录

按以下优先级确定 `$OUTPUT_DIR`：

| 优先级 | 方式 | 示例 |
|--------|------|------|
| 1 | 用户参数明确指定 | `felix-x-writer <文件> --output-dir ~/Desktop/x` |
| 2 | 默认：skill 内部 `output/` 子目录 | `felix-x-writer/output/` |

> 执行前先向用户确认输出目录，未特别说明则用默认目录。

## 执行步骤

### 步骤 0：确定输出目录

```bash
# 默认为 SKILL.md 同级 output/
OUTPUT_DIR="<本 SKILL.md 所在目录>/output"
mkdir -p "$OUTPUT_DIR"
```

### 步骤 1：读取并理解文章

直接读传入的 Markdown 文件（**不复制原文件**）。从 YAML front matter（如有）提取 `title`、`description`。

重点不是"缩写全文"，而是**找出文章里真正值得发 X 的东西**：
- 作者最核心的 1 个判断 / 结论是什么？
- 哪句话最有信息量、最可能引发共鸣或讨论？
- 有没有反常识、有冲突感、或别人会想转发的点？

一篇文章的灵魂往往就 1–3 个点，剩下的都是为了支撑这些点的论据。在 X 上，论据大多可以砍掉。

### 步骤 2：判断生成几条（默认单条，敢于舍弃）

核心原则：**默认就出 1 条**。X 上单条传播力最强，算法也偏爱；thread 会分散互动、增加读者跳出风险。

关键认知：单条不是"把全文所有要点压缩塞进去"——那要么超限，要么挤成一团读着累。单条是**只抓文章里最核心、最值得发的那个判断**，围绕它写满一条（目标权重 240–275），其余次要要点果断舍弃。多数文章的灵魂其实就一句话，单条足够装下。

那什么时候才真的需要 thread？只有当文章天然是**几个并列、各自独立、且都不可舍弃的要点**时——比如"RAG 的三个坑""五个实用技巧"这种列表结构，每个点都是独立的料，单条只选一个会丢掉太多价值。即便如此，也先问自己：能不能挑最强的那个发单条？能就单条。

按以下顺序决策：

| 步骤 | 判断 | 结果 |
|------|------|------|
| 1 | 文章有没有一个最核心的判断，能围绕它写满单条（240–275）？ | **1 条**（默认首选，多数文章走这条） |
| 2 | 没有单一灵魂，而是 2 个并列且都不可舍弃的要点？ | **2 条** |
| 3 | 是 3 个独立、各自完整的要点（列表结构）？ | **3 条**（上限） |

三条铁律：
- **敢于舍弃**：单条优先意味着砍掉次要。一篇讲五个要点的文章，发单条时只取最强的那个，往往比硬塞成拥挤的 thread 更有传播力。别贪心。
- **别浪费额度**：选定了核心判断就把它填到 240–275，别只写一半就停。
- **别为凑额度注水**：填满是"把那个核心判断讲透"，不是塞废话。内容讲完自然停在 200 出头也没问题。

### 步骤 3：生成动态（真诚说理型）

按「风格指南」写。多条时按 thread 结构组织：
- **第 1 条**：放最想说的那句话，是整组动态的灵魂。末尾加 `(1/N)` 标记帮读者预期（N = 总条数）。
- **第 2-N 条**：每条一个独立完整的观点，不要让某条只是上一条的附属。彼此承接但各自能看懂。
- thread 内每条都要能独立成立——X 上很多人只刷到中间一条。

写完后**先不要急着保存**，进步骤 4 校验字数。

### 步骤 4：字数校验（必做，用脚本）

把生成的每条动态，用本 skill 自带的脚本逐条校验。这一步不能省，也不能靠目测——中文字权重是 2，人眼很难算准。

```bash
# 单条校验
python3 scripts/count_chars.py "这里是第1条推文的完整内容……"

# 多条校验（把 thread 写进文件，空行分隔每条）
python3 scripts/count_chars.py -f /tmp/x-thread-draft.txt
```

脚本输出每条的权重和状态（✅ 通过 / ⚠️ 接近上限 / ❌ 超出）。

- 有 ❌ 超出的条目：回去精简——删冗词、砍论据、把长句压短，**或把这一条再拆成两条**（前提是不超过 3 条总量）。
- 重新校验，直到所有条目都通过。
- **目标权重区间 240–275**（约 120–137 中文字）：把单条额度用足，但留约 5–20 权重缓冲，避免微调或加链接时又超。脚本对接近上限的条目标 ⚠️ 属正常，不必非压到 ✅ 安全区。
- **填满 ≠ 注水**：内容讲完自然停在 200 出头也没问题。但如果一条只用了不到 150 权重，先问自己：是不是还有想说的没塞进去？能不能合并成更少的条数？

### 步骤 5：保存输出文件

把最终的动态（含序号、标签）保存到 `$OUTPUT_DIR`。

**文件名格式**：`YYYY-MM-DD_{文章标题}_x.md`（`{文章标题}` 从原文件名提取，`YYYY-MM-DD` 为当天日期）

**文件内容结构**：

```markdown
# {文章标题} - X 动态

> 共 N 条 | 生成于 YYYY-MM-DD
> 原文：{原文件名}

---

## 第 1 条

{第1条推文正文，含 (1/N) 序号和标签}

> 字符数：XXX / 280

---

## 第 2 条

{第2条推文正文}

> 字符数：XXX / 280

---

## 发布说明

- 第 1 条独立发送；若作为 thread，在第 1 条下依次追加回复第 2、3 条。
- {是否带链接、是否@某人等其他提示，没有则省略本段}
```

保存操作：

```bash
BASENAME=$(basename "${INPUT_FILE%.md}")
X_FILE="$OUTPUT_DIR/${BASENAME}_x.md"
# 写入文件内容
```

### 步骤 6：输出结果

直接展示动态内容预览，并说明保存路径与发布提示。

```markdown
## X 动态预览

[第1条内容]

[第2条内容（若有）]

[第3条内容（若有）]

---

📁 已保存至：`$OUTPUT_DIR/YYYY-MM-DD_{文章标题}_x.md`
📝 共 N 条，每条均已通过 280 字符校验
```

## 风格指南（真诚说理型）

这是产出质量的灵魂。每写一句都用下面几条量一下。

### 四把尺子

1. **观点先行** — 第一句就是判断，不是铺垫。X 上滑得快，前 20 个字抓不住人就没了。把"我看了篇文章，觉得……"这种开头全砍掉。
2. **有第一人称、有立场** — 用"我"，敢下判断，敢说"我认为""我的体会是"。别写成正襟危坐的陈述句，也别写成没有主语的客观说明。
3. **克制不浮夸** — 不堆 emoji、不堆感叹号、不"最/第一/碾压/震惊"。一个判断够强，本身就有冲击力，不需要感叹号壮胆。
4. **说人话** — 像 tweet，不像新闻稿。短句、口语、可以带点情绪和幽默。把"综上所述""不可否认"这种废话全删。

### 毛病对照表

| 不要这样写 | 要这样写 |
|-----------|---------|
| 震惊！90% 的人都不知道的 RAG 技巧！🔥🔥🔥 | 做 RAG 半年最大的教训：检索质量比模型选择重要十倍。 |
| 今天给大家分享一篇关于……的文章 | （直接说观点，"分享文章"留给链接本身） |
| 众所周知，AI 正在改变世界…… | 我现在写代码几乎离不开 agent 了，但…… |
| 赋能 / 打通 / 生态闭环 / 沉淀 | 换成具体动作或具体结果 |
| 长句套长句，一句话三四个从句 | 拆成两三个短句，加句号 |

### 标签与互动

- 标签**克制使用**，每条最多 1–2 个，且只放与内容强相关的（`#ClaudeCode` `#RAG` 这种具体领域词）。空泛的 `#AI` `#科技` 少用。
- 不要每条都塞标签。第 1 条抛观点时尤其建议不放，让观点本身先被看到。
- 想引发互动时，结尾可以抛一个**真问题**（不是"你怎么看？"这种万金油），而是具体到能让人想答的问题。不强求，没好问题就别硬加。

### 语言

- **中文为主**。技术专有名词、产品名、代码保留原文（`Claude Code`、`RAG`、`token`），不强翻译。
- 中英混排是 X 技术圈的常态，符合就行。

## 示例

### 示例 A：单条（额度用足）

文章：一篇讲 agent 上下文管理的短文。核心就一个观点，于是尽量填满单条，把观点、对比、做法都讲出来。

**输出**：

> 用 Claude Code 用下来最大感受：agent 的能力上限，不取决于模型多聪明，而取决于你把上下文喂得多干净。塞满无关代码和旧报错的窗口，注意力被稀释，效果能差一个量级；精准裁剪过的窗口，同样任务结果天差地别。喂之前先做减法：删无关文件、清旧报错，只留和当前任务强相关的。#ClaudeCode

（权重 272 / 280，约 124 中文字，落在新目标区间 240–275 内）

### 示例 B：thread（3 条，每条都填满）

文章：一篇讲 RAG 踩坑的长文，有三个独立要点，单条 280 装不下，于是拆成 thread——但每一条都尽量用足额度，不浪费。

**输出**：

> **第 1 条** (1/3)
> 做 RAG 半年，最大的教训：检索质量比模型选择重要十倍。很多人花时间调 embedding、换向量库，却没人认真清语料——去重没做、格式乱、过期信息没清。脏数据进去就是垃圾出来，召回再准也是把错的片段精准送给模型，再强的 LLM 也救不回来。先把语料洗干净，比换什么模型都管用。#RAG
>
> *权重 270 / 280*

> **第 2 条** (2/3)
> 第二个坑：chunk 切得太碎。为了所谓"精准召回"把文档切成 200 字一段，结果是上下文全断——召回的每个片段单看没毛病，拼到一起模型根本串不起完整逻辑，回答支离破碎。chunk 不是越小越准，宁可切大一点、带足上下文，让模型自己判断哪部分有用哪部分该舍，效果比碎切好得多。
>
> *权重 265 / 280*

> **第 3 条** (3/3)
> 第三个坑：只看向量相似度，不管语义结构。向量相近不等于语义相关——经常召回一堆字面像但答非所问的片段。成本低廉的解法：粗筛召回一批后，再加一层 LLM rerank，让模型按"是否真能回答问题"重新打分排序。多一次调用，成本几乎可以忽略，最终回答质量提升非常明显。
>
> *权重 257 / 280*

## 注意事项

- **字数校验不可跳过**——这是本 skill 区别于"凭感觉写"的关键，每条都要过 `count_chars.py`。
- **最多 3 条**。如果文章内容确实需要更多条才能讲清，说明它不适合发 X，提示用户考虑改用公众号或拆成多组动态分开发。
- **默认单条、敢于舍弃**：X 单条传播力远高于 thread。默认只出 1 条——选文章最核心的那个判断，围绕它写满到 240–275，次要要点果断砍掉。只有当文章是"几个并列、各自不可舍弃的独立要点"（如列表技巧）时才拆 thread，且每条同样填满。
- **不替作者编造观点**。动态里的判断必须来自原文，不能为了"有冲击力"加戏。原文没有的结论，不写。
- **保留作者的声音**。如果原文有明显的个人风格（幽默、犀利、温和），动态里要带出来，别改成统一的"AI 推文体"。
- **链接处理**：若用户要在动态里带原文链接，给链接预留 23 权重 + 1 空格，相应减少正文中文字数。校验脚本会自动把 URL 算作 23。
- 技术类内容保持专业性，不过度娱乐化；非技术内容也守"克制不浮夸"的底线。
