# 文章转小红书文案

将本地 Markdown 文章转换为小红书风格文案，并生成可直接作为图片生成提示词的封面图要点文件和信息图。所有文件存储到同一目录，保持内容精华的同时适配小红书的内容呈现方式。

## 参数

- `$ARGUMENTS`: 本地 Markdown 文件路径（必填），以及可选的输出目录路径

## 输出目录

输出目录按以下优先级确定（从高到低）：

| 优先级 | 方式 | 示例 |
|--------|------|------|
| 1 | 用户在参数中明确指定目录 | `/felix-xhs-writer <文件> --output-dir ~/Desktop/xhs` |
| 2 | 默认：输入 Markdown 文件所在目录 | 输入 `draft/article.md` 则输出到 `draft/` |

> 若用户已经明确指定输出目录，直接使用指定目录；否则不要追问，默认输出到输入 Markdown 文件所在目录。

## 执行步骤

### 步骤 0：确定输出目录

根据上方优先级规则确定 `$OUTPUT_DIR`：
- 若用户指定了目录，使用用户指定的路径（支持 `~` 和相对路径）
- 否则默认为：输入 Markdown 文件所在目录，即 `dirname "$INPUT_FILE"`。例如输入 `draft/article.md`，输出文件也保存到 `draft/`。

确保目录存在：
```bash
if [ -z "$OUTPUT_DIR" ]; then
  OUTPUT_DIR="$(dirname "$INPUT_FILE")"
fi
mkdir -p "$OUTPUT_DIR"
```

### 步骤 1：读取文章内容

将直接使用传入的本地 Markdown 文件，**不进行复制**。

读取 Markdown 文件，从 YAML front matter 中提取 `title`、`author`、`description` 等元信息。

```bash
SAVED_MD="$INPUT_FILE"
```

### 步骤 2：分析文章结构

提取文章的核心要素：
- **标题**: 吸引眼球的亮点
- **痛点**: 问题或需求
- **核心内容**: 3-5 个关键要点
- **价值**: 读者能获得什么

### 步骤 3：转换文案风格

按照小红书内容特点进行转换：

#### 结构模板

```markdown
# 🔥 吸引眼球的标题（使用emoji）

✨ 开头钩子（引起共鸣）
✨ 提出问题
✨ 引出下文


## 🎯 核心内容

### 1️⃣ 要点一
- 关键信息
- 实用技巧

### 2️⃣ 要点二
- 关键信息
- 实用技巧

### 3️⃣ 要点三
- 关键信息
- 实用技巧


## 💡 实用总结/行动建议

#话题标签1 #话题标签2 #话题标签3
```

#### 文案风格要求

1. **标题**: 🔥 emoji + 痛点/亮点 + 情绪词
2. **开头**: 3-5 个钩子句，使用 ✨ emoji
3. **正文**:
   - 分点阐述，每点使用 emoji 数字标记
   - 简洁有力，避免冗长
   - 重点内容加粗或使用特殊符号
4. **结尾**: 简洁总结 + 话题标签

### 步骤 4：保存小红书文案

将生成的小红书文案保存到 `$OUTPUT_DIR` 目录中，严格控制在1000字以内。

**文件名格式**: `YYYY-MM-DD_{文章标题}_xhs.md`（基于原文件名生成）

**文件内容结构**:
```markdown
# 小红书文案标题

[文案内容...]
```

> 注意：不添加 YAML frontmatter 元信息

**保存操作**:
```bash
# 提取原文件名（不含扩展名和路径）
BASENAME=$(basename "${SAVED_MD%.md}")

# 统一在 OUTPUT_DIR 目录下生成小红书文案
XHS_FILE="$OUTPUT_DIR/${BASENAME}_xhs.md"

# 写入文件
cat > "$XHS_FILE" << 'EOF'
$XHS_CONTENT
EOF
```

### 步骤 5：生成可直接用于图片生成的封面图要点文件

从小红书文案中提取核心要点，生成 `$COVER_FILE`。这个文件不是普通备注，而是**图片生成提示词的唯一来源**：后续生成信息图时，直接把该文件中的“图片生成提示词”段落喂给图片生成工具，不再另起一份 prompt。

#### 5.1 分析内容提取要点

从小红书文案中提取：
- **主题**: 文章核心主题（标题）
- **副标题**: 补充说明或价值主张
- **画面文字**: 希望图片中出现的主标题、副标题、卡片标题和短句
- **核心要点**: 3-6 个关键点，优先短句，避免塞入长段落
- **一句话总结**: 核心原则或行动建议

#### 5.2 封面图要点文件格式

`cover-points.md` 必须写成“人能读、模型也能直接用”的格式，包含完整图片生成约束：

```markdown
# {主题} - 封面图要点

## 图片生成提示词
请生成一张小红书竖版信息图，比例 3:4。

主题：{主题}
副标题：{副标题}

画面文字请尽量使用以下中文原文，保证清晰可读，不要错字，不要小字堆叠：
主标题：{主标题}
副标题：{副标题}

核心要点区块：
1. {要点1，短句}
2. {要点2，短句}
3. {要点3，短句}
4. {要点4，短句，可选}
5. {要点5，短句，可选}

视觉风格：
- Notion 极简风格，手绘线条，纸张质感
- 柔和温暖配色，不刺眼
- 主标题大且醒目，副标题简短
- 3-6 个信息区块，层级清晰，留白充足
- 专业、可信赖、真实分享

排版建议：
- 顶部大标题，中部放一个与主题相关的简洁手绘视觉元素
- 下方或两侧排列核心要点卡片，每张卡片配一个线性小图标
- 中文文字必须清晰可读，避免过密排版

限制：
- 不要二维码、logo、水印、真实人像
- 不要添加与文章无关的主题或品牌

## 封面展示要点

| # | 要点 |
|---|------|
| 1 | {要点1} |
| 2 | {要点2} |
| ... | ... |

## 一句话总结
{一句话总结}
```

#### 5.3 保存封面图要点文件

**文件名格式**: `YYYY-MM-DD_{文章标题}_cover-points.md`

**保存操作**:
```bash
# 统一在 OUTPUT_DIR 目录下生成封面图要点/图片提示词文件
COVER_FILE="$OUTPUT_DIR/${BASENAME}_cover-points.md"
# 写入封面图要点文件
```

### 步骤 6：基于封面图要点文件生成信息图

必须使用 Codex 当前会话可用的图片生成工具，基于 `$COVER_FILE` 生成一张小红书竖版信息图。

**硬性要求**：
- 必须调用 Codex 图片生成能力（例如当前运行时提供的 `image_gen` / `imagegen` 工具）。
- 禁止用本地绘图库、Python/Pillow、Canvas、HTML 截图、SVG 手绘、脚本拼图、第三方图片服务或其他非 Codex 图片生成方式替代。
- 若当前环境没有可用的 Codex 图片生成工具，不得生成兜底图片；保留 `$COVER_FILE`，并在最终结果中明确说明“信息图未生成：当前环境没有可用的 Codex 图片生成工具”。

#### 6.1 使用提示词

- 直接读取 `$COVER_FILE` 的 `## 图片生成提示词` 段落作为 Codex 图片生成 prompt。
- 不要再生成独立的 `cover-infographic-prompt.md`，除非用户明确要求备份 prompt。
- 如需微调 prompt，只允许做格式整理或补充图片工具所需的技术字段；不得增删核心主题、核心要点和限制项。

#### 6.2 生成并保存图片

- 使用 Codex 当前可用的图片生成工具生成 PNG 信息图；这是唯一允许的图片生成来源。
- 生成内容必须严格基于 `$COVER_FILE`，不额外发散无关主题。
- 若生成结果包含明显错字、文字过密、要点缺失、主题偏离、二维码、logo、水印或真实人像，必须重新生成一次。
- 将 Codex 生成的最终图片复制到 `$OUTPUT_DIR`，保留 Codex 默认生成目录中的原始文件，不要移动或删除原图。
- 禁止在 Codex 生成失败后改用本地脚本生成同名兜底图片。

**文件名格式**: `YYYY-MM-DD_{文章标题}_cover-infographic.png`

**保存操作**:
```bash
INFOGRAPHIC_FILE="$OUTPUT_DIR/${BASENAME}_cover-infographic.png"
# 将 Codex 生成的最终图片复制到该路径，保留原始生成文件
```

若当前环境没有可用的 Codex 图片生成能力，则保留 `$COVER_FILE` 作为可直接复制使用的图片生成提示词，并在最终结果中明确说明未生成图片；不要创建 `$INFOGRAPHIC_FILE`。

### 步骤 7：输出结果

直接展示小红书文案内容预览，并在结尾说明文件保存路径。

**输出格式**:
```markdown
## 小红书文案预览

[小红书文案内容...]

---

## 封面图要点 / 图片提示词预览

[封面图要点 / 图片提示词内容...]

---

## 信息图

[信息图文件路径或未生成原因]

---

📁 文件已保存至:
- 小红书文案: `$OUTPUT_DIR/YYYY-MM-DD_{文章标题}_xhs.md`
- 封面图要点 / 图片提示词: `$OUTPUT_DIR/YYYY-MM-DD_{文章标题}_cover-points.md`
- 封面信息图: `$OUTPUT_DIR/YYYY-MM-DD_{文章标题}_cover-infographic.png`
```

## 内容转换原则

### 精简原则
- 删除冗余表述，保留核心信息
- 每个要点不超过 3 句话
- 使用短句和短语，提高可读性

### 情感化表达
- 使用"宝子们"、"家人们"等亲切称呼
- 添加"超赞"、"必看"、"神器"等情绪词
- 使用感叹号增强语气

### 视觉优化
- 合理使用 emoji 分隔内容
- 使用引用块、列表等格式
- 段落之间保持适当间距

## 常用 emoji 组合

- **开头**: ✨ 🔥 💡 📢 🎯
- **要点**: 1️⃣ 2️⃣ 3️⃣ 4️⃣ 5️⃣
- **强调**: 💪 ⚡ 🌟 💎 🚀
- **总结**: 📝 ✅ 💡 🎁

## 默认文件存储结构

```
输入文件所在目录/
├── 原文章.md
├── YYYY-MM-DD_{文章标题}_xhs.md                # 小红书文案
├── YYYY-MM-DD_{文章标题}_cover-points.md       # 封面图要点 / 图片生成提示词
└── YYYY-MM-DD_{文章标题}_cover-infographic.png # 封面信息图
```

> 注：`YYYY-MM-DD` 为当前日期，`{文章标题}` 从文件名自动提取

## 注意事项

- 保持原内容的核心价值不变
- 字数严格控制在 1000 字以内
- 标签要与内容高度相关
- 避免过度营销化，保持内容真实
- 技术类内容保持专业性，不过度娱乐化
- **输出目录**：默认与输入 Markdown 文件所在目录保持一致；可在调用时通过 `--output-dir` 参数自定义
- **信息图生成**：必须通过 Codex 图片生成工具完成；不得使用本地脚本或其他方式生成替代图片
